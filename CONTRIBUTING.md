# Contributing to FieldSpace Protocol

Thank you for your interest in contributing to the FieldSpace Protocol specification!

## How to Contribute

### Reporting Issues

- Use GitHub Issues to report problems with the specification
- Include specific section references when reporting ambiguities
- Provide examples that illustrate the issue

### Proposing Changes

1. **Open an Issue First** — For significant changes, open an issue to discuss before submitting a PR
2. **Fork the Repository** — Create your own fork to work on changes
3. **Create a Branch** — Use a descriptive branch name (e.g., `clarify-capability-types`)
4. **Make Changes** — Follow the style guidelines below
5. **Submit a Pull Request** — Reference any related issues

### Style Guidelines

#### Markdown

- Use ATX-style headers (`#`, `##`, etc.)
- Use fenced code blocks with language identifiers
- Keep lines under 100 characters where practical
- Use tables for structured comparisons

#### JSON Examples

- Use 2-space indentation
- Include comments explaining non-obvious fields
- Ensure all examples validate against the schema

#### Terminology

- Use consistent terminology as defined in `concepts.md`
- "Provider" not "server" or "service"
- "Client" not "consumer" or "app"
- "Capability" not "feature" or "function"

## Versioning

The protocol follows [Semantic Versioning](https://semver.org/):

- **MAJOR** — Breaking changes to the protocol
- **MINOR** — Backwards-compatible additions
- **PATCH** — Clarifications and corrections

## Code of Conduct

Be respectful and constructive. We're building an open standard for everyone.

## Questions?

Open a Discussion on GitHub if you have questions about contributing.
